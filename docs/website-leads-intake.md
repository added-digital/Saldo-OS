# Website → OS lead intake

The public marketing site forwards contact-form submissions to OS. OS is the
lead store and the sender: each submission is persisted (visible in **Leads** in
the dashboard) and a notification email is sent to the owning inbox via the
Azure app-only Microsoft Graph flow. The site may keep writing to its CMS as a
fallback — OS is the system of record but need not be the only copy.

The site already runs anti-bot (honeypot + time trap) and reCAPTCHA before
forwarding, so OS does **not** re-screen. It does validate structure and
enforce idempotency.

## 1. Endpoint

```
POST https://os.saldoredo.se/api/leads/intake
Content-Type: application/json
```

## 2. Auth

Shared bearer token in the `Authorization` header:

```
Authorization: Bearer <LEADS_INTAKE_TOKEN>
```

Compared in constant time against the `LEADS_INTAKE_TOKEN` env var on OS.
Missing/invalid → `401`. Rotate by changing the env var on both sides.

## 3. Request payload

```json
{
  "formName": "contact",
  "fields": {
    "name": "",
    "email": "",
    "company": "",
    "phone": "",
    "message": ""
  },
  "pagePath": "/kontakt",
  "submittedAt": "2026-06-29T10:00:00+02:00",
  "meta": { "recaptchaScore": 0.9 }
}
```

| Field              | Required | Notes |
|--------------------|----------|-------|
| `fields.name`      | yes      | 1–200 chars |
| `fields.company`   | yes      | 1–200 chars |
| `fields.message`   | yes      | 1–5000 chars |
| `fields.email`     | no       | valid email if present; `""`/omitted/`null` all accepted. When present, becomes the notification email's **Reply-To** |
| `fields.phone`     | no       | up to 50 chars |
| `formName`         | no       | defaults to `"contact"` |
| `pagePath`         | no       | up to 500 chars |
| `submittedAt`      | no       | ISO-8601 with offset |
| `meta`             | no       | free-form object; `meta.recaptchaScore` (number) is stored on the lead |

## 4. Behaviour on receipt

1. Validate auth, JSON, and the schema above.
2. Honour the optional `Idempotency-Key` header (see §6).
3. Store the lead via the service-role client (`website_leads` table) — it
   appears immediately in the **Leads** view.
4. Send a notification email via Azure app-only Graph to the configured
   recipient inbox(es), with the submitter's email as **Reply-To** when present.

### Recipient

Comma-separated inbox list from the `LEADS_NOTIFICATION_EMAIL` env var on OS —
configurable without code changes. Seeded with
`matias.a@saldoredo.se,sofia@saldoredo.se,johan@saldoredo.se`. Sent from the
neutral mailbox `noreply@saldoredo.se` (`GRAPH_MAIL_SENDER`), never a person's
mailbox. This is the only thing that decides who gets notified for now; per-form
routing can be added later behind the same endpoint without changing the site
contract.

### Graceful degradation

The email send is **best-effort and never fails the request**. If the Azure
app-only mail isn't configured yet (admin consent / sender mailbox pending) or
no recipient is set, the lead is still stored and the notification is recorded
as `skipped`. A Graph failure is recorded as `failed`. Either way the site gets
a success response and no lead is lost. Email "lights up" automatically once
consent + env vars are in place — no site change, no redeploy of the site.

## 5. Response contract

Success (new lead) → **201**:

```json
{ "ok": true, "leadId": "uuid", "duplicate": false, "notificationStatus": "sent" }
```

`notificationStatus` ∈ `sent | failed | skipped`. Duplicate (idempotent replay)
→ **200** with `"duplicate": true`.

| Status | `error` | Meaning |
|--------|---------|---------|
| `400`  | `invalid_body` | empty body or invalid JSON |
| `401`  | `unauthorized` | missing/invalid bearer token |
| `422`  | `validation_error` | schema failed; `issues: [{ path, message }]` |
| `500`  | `server_error` | failed to store the lead |

The site should treat `401`/`422` as non-retryable (fix and resubmit) and
`500`/network errors as retryable (resend with the same `Idempotency-Key`).

## 6. Idempotency & validation

**Idempotency (recommended).** Send a stable, unique key per submission:

```
Idempotency-Key: <opaque string, e.g. a UUID generated by the site>
```

A repeat POST with a key already seen returns the original lead (`200`,
`duplicate: true`) and does **not** create a duplicate or re-send email. Without
a key, retries can create duplicate leads — so send one if the site retries on
timeout. Enforced by a unique index on `website_leads.idempotency_key`.

**Server-side validation enforced:** bearer auth; JSON well-formedness; required
`name`/`company`/`message`; email format when present; field length caps; ISO
datetime format for `submittedAt`. Anti-bot/reCAPTCHA are **not** re-run.

## Env vars on OS

| Var | Purpose | Value |
|-----|---------|-------|
| `LEADS_INTAKE_TOKEN` | shared bearer secret (site sends it) | _set in Vercel + `.env.local`; share out-of-band, not in git_ |
| `LEADS_NOTIFICATION_EMAIL` | recipient inbox(es), comma-separated | `matias.a@saldoredo.se,sofia@saldoredo.se,johan@saldoredo.se` |
| `GRAPH_MAIL_SENDER` | neutral mailbox the notification sends from (UPN) | `noreply@saldoredo.se` |
| `AZURE_AD_TENANT_ID` / `AZURE_AD_CLIENT_ID` / `AZURE_AD_CLIENT_SECRET` | app-only Graph auth | reused (already set) |

The website only needs the URL (`https://os.saldoredo.se/api/leads/intake`) and
the `LEADS_INTAKE_TOKEN` above. The other vars live only on OS.

### One-time Azure prerequisite (not a code step)

App-only sending needs an Azure admin to grant the app registration the
**application** permission `Mail.Send` (Microsoft Graph) **with admin consent**,
and a sender mailbox set in `GRAPH_MAIL_SENDER`. This can be added to the same
app registration that already powers the delegated login flow. Until that lands,
lead capture works and notifications are marked `skipped`.
